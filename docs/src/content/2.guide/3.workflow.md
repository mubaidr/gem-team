## Phase Details

### Phase 0: Init & Clarify

The Orchestrator assesses every incoming request:

1. **Reads all context** — external refs, error context, user config (`.gem-team.yaml`)
2. **Detects intent** — feature, bug-fix, refactor, docs, research, etc.
3. **Resolves Plan ID** — continues existing plan or starts new one
4. **Classifies complexity** — TRIVIAL / LOW / MEDIUM / HIGH
5. **Identifies gray areas** — documents assumptions, asks only for blocking clarifications

### Phase 1: Route

Based on the assessment, routes to the appropriate path:

- `continue_plan` + no feedback → load plan → Phase 3
- `continue_plan` + feedback → load plan → Phase 2
- `new_task` → Phase 2

### Phase 2: Planning

Adapted to complexity:

- **TRIVIAL/LOW**: Orchestrator creates a minimal ephemeral plan directly.
- **MEDIUM**: Delegates to `gem-planner`, then `gem-reviewer` validates the plan.
- **HIGH**: Same as MEDIUM, plus `gem-critic` reviews architecture, contracts, and breaking changes.

If validation fails: replan with findings, or escalate if not replanable.

### Phase 3: Execution

Work is executed in **waves** — groups of tasks that can run in parallel:

1. Each wave collects unblocked pending tasks
2. Tasks are delegated to the appropriate specialist agent
3. **Integration gates** verify output:
   - HIGH complexity: reviewer checks every wave
   - MEDIUM: gates on risk (final wave always gated)
4. High-confidence learnings are persisted after each wave

### Phase 4: Output

A structured status report showing:

- Plan ID and objective
- Task completion progress (completed/total, percentage)
- Wave status
- Blocked tasks with reasons
- Next steps
